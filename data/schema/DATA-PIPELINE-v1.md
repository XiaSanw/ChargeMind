# ChargeMind 数据清洗方案 v1

> 目标：将多源原始数据合并为统一的场站画像，供 RAG 知识库和算法 Stub 使用
> 日期：2026-04-23

---

## 一、输入数据源

| 文件 | 格式 | 记录数 | 说明 |
|------|------|--------|------|
| `表1.xlsx` | Excel | 13,456 | 最全场站静态信息（装机功率、土地类型、服务车型等） |
| `b2.csv` | CSV | 14,711 | 场站基础信息 + 分时电价/服务费 + 营业时间 |
| `b4.csv` | CSV | 13,290 | 各功率段**装机功率**分布 |

## 二、输出格式

### `data/cleaned/stations.jsonl`
每行一个 JSON 对象，代表一个场站的完整画像。

核心字段：
- `station_id`, `station_name`, `charger_type`（直流/交流）
- `operator_id`, `lng`, `lat`, `region`（行政区）
- `total_installed_power_kw`（装机功率 kW）, `land_property`（土地性质）, `is_public`（是否对外开放）
- `service_car_types`（服务车型列表）
- `station_status`, `station_status_desc`（状态描述）, `business_hours`（营业时间）
- `electricity_fee` / `service_fee`（结构化分时电价，含均价/最低/最高）
- `power_structure`（功率段分布：小于30kW、30-120kW、120-360kW、大于等于360kW）
- `avg_daily_energy_kwh`（日均充电量，度）
- `avg_utilization`（平均利用率）
- `peak_hour`（功率高峰时段）, `valley_hour`（功率低谷时段）
- `season_stats`（夏季/冬季/春节分季统计）
- `has_timeseries_data`（是否有真实时序数据）
- `data_quality_score`（数据质量评分 0-1）

### `data/cleaned/stations_summary.csv`
扁平化摘要表，方便快速查看和人工核验。

## 三、详细处理步骤

### 步骤1：读取并清洗 `表1.xlsx`

**源 Sheet**：`biao1`（13,456 行，14 列）

| 原始字段 | 清洗操作 | 输出字段 |
|---------|---------|---------|
| `operator_id` | 转为字符串，去除空格 | `operator_id` |
| `station_id` | 转为字符串，去除空格 | `station_id` |
| `station_name` | 去除首尾空格 | `station_name` |
| `station_lng` | 转为浮点数 | `lng` |
| `station_lat` | 转为浮点数 | `lat` |
| `total_installed_power` | 转为浮点数（单位：kW） | `total_installed_power_kw` |
| `land_property` | 码表映射（见下文） | `land_property` |
| `is_public` | 1→true, 0→false, 其他→null | `is_public` |
| `station_status` | 保留原始码值 | `station_status` |
| `service_car_types` | 按逗号拆分数字，码表映射 | `service_car_types` |
| `le_30kw_count` | 转为整数 | `gun_count_lt_30kw` |
| `gt_30_le_120kw_count` | 转为整数 | `gun_count_30_120kw` |
| `gt_120_le_360kw_count` | 转为整数 | `gun_count_120_360kw` |
| `gt_360kw_count` | 转为整数 | `gun_count_gte_360kw` |

**码表映射：**

```python
土地性质映射 = {
    1: "国有用地",
    2: "集体用地",
    3: "私有用地",
    4: "租赁用地",
    10: "商业用地",
    255: "其他"
}

服务车型映射 = {
    1: "公交车", 2: "出租车", 3: "物流车", 4: "通勤车",
    5: "大巴车", 6: "私家车", 7: "环卫车", 8: "泥头/重卡车",
    9: "公务车", 10: "网约车", 11: "港口码头作业车", 255: "其他"
}

场站状态映射 = {
    5: "运营中",
    50: "运营中",
}
```

**空值处理：**
- `land_property` 为空 → 标记为 "未知"
- `service_car_types` 为空 → 标记为 ["其他"]
- `total_installed_power` 为空或 ≤0 → 尝试从 b4 的 total_power 填充


### 步骤2：读取并清洗 `b2.csv`

**字段**：operator_id, station_id, station_name, station_lng, station_lat, electricity_fee, service_fee, station_status, busine_hours

**清洗操作：**

| 字段 | 操作 |
|------|------|
| 所有ID | 转为字符串，去除引号和空格 |
| `station_name` | 从后缀提取 `charger_type`（直流/交流）；保留清洗后的名称 |
| `electricity_fee` | 解析为结构化时段（见下文解析器） |
| `service_fee` | 解析为结构化时段 |
| `busine_hours` | 为空则填充为 "00:00~24:00" |
| `station_status` | 转为整数，保留原值 |

**分时电价解析器：**

输入格式有两种：
- 简单数字：`"3.0000"`、`"0.6"`
- 时段字符串：`"00:00~07:00,0.230000;07:00~09:00,0.670000;..."`

解析逻辑：
1. 为空字符串 → 返回 None
2. 不含 `~` 和 `;` → 简单数字，生成单一时段 00:00~24:00
3. 含时段 → 按 `;` 拆分，每段按 `,` 拆分为时间范围和价格
4. 时间范围按 `~` 拆分为起始和结束时间
5. 计算所有时段的均价、最低价、最高价

已知问题处理：
- 部分时段不连续（如 10:00~11:00 后直接 12:00~14:00，缺 11:00~12:00）
- 保留原样，在 data_quality_notes 中记录时段缺口

### 步骤3：合并 表1 + b2

- 基准表：表1（13,456 条，静态字段更完整）
- 以 `station_id` 为键左连接 b2
- 冲突字段（station_name, lng, lat, station_status）以表1为准
- b2 补充：electricity_fee, service_fee, busine_hours（表1没有这些字段）
- 从 b2 的 station_name 提取 charger_type

结果：约 13,000 条基础记录，带上了电价和营业时间。

### 步骤4：读取并清洗 `b4.csv`

**字段**：operator_id, station_id, power_lt_30kw, power_30_120kw, power_120_360kw, power_gte_360kw, total_power

**关键发现**：b4 的联合主键是 `(operator_id, station_id)`。同一个 station_id 可能对应多个 operator_id。

**清洗操作：**
- 所有功率值转为浮点数
- 重复 station_id 的处理：按 station_id 分组，对各功率段**求和**（反映场站总装机）
- 验证：total_power 应等于各功率段之和

字段映射：
- power_lt_30kw → power_structure.lt_30kw_kw
- power_30_120kw → power_structure._30_120kw_kw
- power_120_360kw → power_structure._120_360kw_kw
- power_gte_360kw → power_structure.gte_360kw_kw

### 步骤5：合并 + b4

- 以 `station_id` 为键左连接 b4（已按 station_id 聚合）
- 如果表1的 total_installed_power_kw 为空，尝试用 b4 的 total_power 填充

结果：约 13,000 条记录，带上了功率结构。


### 步骤6：处理 `result_power_by_slot.csv`

**数据量**：233万行，30天，7,180个场站，小时粒度

**字段**：运营商编号, 场站编号, 日期, 时段, 各功率段枪的总功率, 场站总功率

**按场站聚合计算：**

1. **日均充电量（度）**
   - 一天的 24 个小时功率加起来 = 日充电量（kW × 1小时 = kWh = 度）
   - `avg_daily_energy_kwh` = 所有天数的日充电量均值

2. **平均利用率**
   - 每小时利用率 = 场站总功率 / total_installed_power_kw
   - `avg_utilization` = 所有小时利用率的均值
   - 注：需要用到步骤5合并后的 total_installed_power_kw
   - 如果装机功率为空或0，跳过利用率计算

3. **高峰/低谷时段**
   - 按小时分组（00:00, 01:00, ..., 23:00）
   - 计算每个小时在所有天内的平均功率
   - `peak_hour` = 平均功率最高的小时
   - `valley_hour` = 平均功率最低的小时

4. **分季统计**
   - 春节：2025-01-27 至 2025-02-04
   - 夏季：2025-06-30 至 2025-07-07
   - 国庆：2025-09-29 至 2025-10-03
   - 冬季：2025-12-15 至 2025-12-22
   - 每季计算：avg_utilization, avg_daily_energy_kwh

5. **覆盖标记**
   - `has_timeseries_data` = True（该场站出现在 result_power_by_slot 中）
   - 约 6,000 个场站为 False

**性能考虑：**
- 文件 147MB，233万行，pandas 可直接读入内存
- 如内存紧张，可分块读取
- 可先过滤为静态数据集中存在的场站

### 步骤7：合并 + 时序指标

- 以 `station_id` 为键左连接时序聚合结果
- 约 7,180 个场站获得真实指标
- 约 6,000 个场站指标为空（在步骤8填充）

### 步骤8：区域均值填充（方案A）

对于没有时序数据的场站：

1. **判定场站所属行政区**
   - 首选：场站名称关键词匹配（含"南山"→南山区，"福田"→福田区等）
   - 备选：经纬度反地理编码（API 可能有速率限制）
   - 兜底：标记为"深圳市"

2. **计算区域均值**（从有时序数据的场站中统计）
   - 按行政区分组
   - 计算：avg_daily_energy_kwh, avg_utilization, peak_hour（众数）, valley_hour（众数）

3. **填充无数据场站**
   - 查找该场站的行政区
   - 用区域均值填充 avg_daily_energy_kwh, avg_utilization, peak_hour, valley_hour
   - `has_timeseries_data` = False
   - 增加标记：`metrics_estimated_from_regional_avg` = true

4. **如果行政区无法判定**
   - 用全市均值作为兜底

### 步骤9：计算数据质量评分

每个场站的评分 = 加权求和：
- 有静态信息（表1）：+0.3
- 有电价数据（来自b2）：+0.2
- 有功率结构（来自b4）：+0.15
- 有真实时序数据：+0.25
- 时序数据为估算值：+0.10
- 缺失关键字段（名称、位置、功率）：每项 -0.2

评分范围 0~1，用于 RAG 检索时优先返回高质量场站。


### 步骤10：输出

**主输出：`data/cleaned/stations.jsonl`**
- 每行一个 JSON 对象
- UTF-8 编码
- 约 13,000 条记录

**辅助输出：`data/cleaned/stations_summary.csv`**
- 关键字段的扁平化表格
- 列：station_id, station_name, region, total_installed_power_kw, avg_daily_energy_kwh, avg_utilization, has_timeseries_data, data_quality_score

**报告输出：`data/cleaned/pipeline_report.json`**
- 处理统计
- 数据覆盖率
- 质量分布
- 问题日志

   - 无法判定的标记为 "深圳市"

## 九、周边业态与人流热力分析（新增）

### 目标

不依赖实时 API 查询和交通流数据，通过**场站名称语义识别** + **抽样 POI 验证**，给每个场站打上"周边业态"标签，用于推断人流热力类型。

### 输入

- 已清洗的场站数据（含 station_name, lng, lat）
- 高德 POI 搜索 API（仅用于抽样验证，100-200 个场站）

### 处理步骤

#### 步骤11：名称关键词推断周边业态

从 `station_name` 提取关键词，匹配业态类型：

| 业态类型 | 关键词规则 | 人流特征 |
|---------|-----------|---------|
| 住宅区 | 含"小区""花园""家园""公寓""村""苑" | 居住型人流，晚间充电需求高 |
| 办公区 | 含"大厦""中心""广场""科技园""写字楼""工业园" | 办公型人流，白天充电需求高 |
| 商业区 | 含"商场""购物中心""MALL""商业街""广场" | 消费型人流，随机停留充电 |
| 交通枢纽 | 含"地铁站""地铁""公交总站""车站""机场""港口" | 换乘型人流，短时补电 |
| 工业区 | 含"工厂""工业区""产业园""物流园""仓库" | 物流/营运车为主 |
| 旅游景区 | 含"公园""景区""酒店""度假村" | 旅游车流，季节性波动 |
| 混合区域 | 同时命中住宅+办公/商业 | 综合型人流，全天均衡 |
| 未知 | 未命中任何关键词 | 需要 POI 补充或人工标注 |

**匹配逻辑：**
- 一个场站可能同时命中多个业态（如"XX科技园购物中心充电站"→办公+商业）
- 优先级：交通枢纽 > 商业区 > 办公区 > 住宅区 > 工业区 > 旅游景区
- 输出 `surrounding_business_types` 为列表，按优先级排序

#### 步骤12：推断人流热力类型

基于周边业态，综合判断人流热力类型：

```
人流热力类型 = f(周边业态, 服务车型, 营业时间)
```

| 周边业态 | 服务车型 | 营业时间 | 推断人流热力类型 |
|---------|---------|---------|----------------|
| 住宅区 | 私家车 | 00:00~24:00 | 居住型-夜间高峰 |
| 办公区 | 通勤车/私家车 | 07:00~19:00 | 办公型-日间高峰 |
| 商业区 | 混合 | 10:00~22:00 | 消费型-午间/晚间高峰 |
| 交通枢纽 | 出租车/网约车 | 00:00~24:00 | 换乘型-全天波动 |
| 工业区 | 物流车/营运车 | 06:00~22:00 | 工业型-早晚双峰 |
| 混合区域 | 混合 | 00:00~24:00 | 综合型-全天均衡 |

输出字段：
- `foot_traffic_heat_type`: 人流热力类型标签
- `peak_demand_period`: 推断的充电需求高峰时段
- `demand_characteristics`: 需求特征描述（如"以私家车夜间慢充为主"）

#### 步骤13：抽样 POI 验证（可选，Demo 阶段不做全量）

为验证名称推断的准确率，抽样 100-200 个场站：

1. 随机抽取不同业态类型的场站各 20-30 个
2. 调用高德 POI 搜索 API（半径 500 米）
3. 查询 POI 类型：商场、写字楼、住宅区、地铁站
4. 对比名称推断结果 vs POI 实际结果
5. 计算准确率，生成验证报告

**验证报告内容：**
- 总体准确率
- 各业态类型的准确率
- 误判案例分析
- 关键词规则优化建议

### 与诊断流程的结合

周边业态标签在诊断中的使用：

1. **算法 Stub 输入**：作为场站特征之一，影响收益预测
   - 办公区 → 预测白天高利用率
   - 住宅区 → 预测夜间高利用率
   - 商业区 → 预测随机性充电，周转率低但客单价高

2. **RAG 检索维度**：查找相似案例时，优先匹配同业态类型场站
   - 用户诊断一个"科技园充电站"
   - RAG 优先检索"办公区"场站的案例

3. **报告输出**：诊断报告中展示"周边业态分析"段落
   - "该场站位于办公区，周边有XX个写字楼，主要服务通勤车辆..."
   - 结合分时电价建议"建议设置日间优惠套餐吸引上班族..."
