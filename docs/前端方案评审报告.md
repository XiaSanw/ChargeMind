### 3.2 技术架构评审

#### 3.2.1 状态管理

**方案现状**：
- `/input` → `/enrich`：通过 URL query 参数传递 `profile`
- `/enrich` → `/report`：通过 URL query 参数传递 `result`

**风险分析**：
- `profile` 对象包含多个字段（region, business_type, total_installed_power, pile_count, monthly_rent, staff_count, avg_price），序列化后 URL 长度约 200-500 字符，未超浏览器限制（~2000字符）。
- `result` 对象包含 `algorithm + rag + report`，其中 `rag.analysis` 为 Markdown 格式 LLM 分析报告，长度可能达 **3000-8000 字符**，**远超 URL 长度限制**。

**评审结论**：⚠️ **URL 传 result 不可行，必须调整**。

| 方案 | 优点 | 缺点 | 建议 |
|------|------|------|------|
| **A. localStorage** | 无长度限制，刷新不丢失 | 跨域/隐私模式下可能受限 | **推荐用于 profile**，简单可靠 |
| **B. sessionStorage** | 同 localStorage，关闭标签页清空 | 刷新页面数据丢失 | 不推荐 |
| **C. Context + 内存** | 无长度限制，React 原生 | 刷新页面数据丢失 | **推荐用于 result**，配合 diagnose 后跳转，数据在内存中传递 |
| **D. 后端缓存** | 最可靠，支持分享链接 | 需额外开发 session 机制 | Demo 阶段过度设计 |

**推荐策略**：
- `profile`：localStorage 持久化（支持问卷填写到一半刷新不丢失）。
- `result`：React Context 或全局 state（如 Zustand）内存传递，不经过 URL。报告页刷新后可通过重新调用 diagnose 恢复（成本可接受）。

#### 3.2.2 组件复用

**方案现状**：三页面独立开发（`StationInputPage.tsx` / `EnrichPage.tsx` / `ReportPage.tsx`）。

### 3.3 后端接口对接评审

#### 3.3.1 接口清单与前端职责

| 接口 | 方法 | 前端输入 | 前端输出处理 | 延迟预期 |
|------|------|---------|------------|---------|
| `/api/extract` | POST | `user_input: string` | 解析成功 → 跳转 enrich；失败 → 提示重试 | < 2s |
| `/api/enrich` | POST | `profile: object` | 获取 `next_question`，渲染卡片；`complete=true` 时调用 diagnose | < 1s |
| `/api/diagnose` | POST | `profile: object` | 获取 algorithm + rag + report，渲染报告页 | **5-10s** |

#### 3.3.2 关键对接细节

**`/api/extract` 对接**：
- 输入框需防 XSS（用户输入直接提交后端，前端不渲染原始 HTML）。
- 提交按钮需防重复提交（loading 状态锁定）。
- 返回的 `profile` 字段可能不完整（如用户输入"我在南山建了个充电站"，只解析出 region），属于预期行为，直接进 enrich。

**`/api/enrich` 对接**：
- 循环调用：每回答一题 → 更新 profile → POST enrich → 获取下一题。
- 需维护本地 profile state，支持"跳过"时填充默认值。
- 建议问题类型映射到 shadcn/ui 组件：
  - `type: "text"` → `Input`
  - `type: "number"` → `Input type="number"`
  - `type: "select"` → `Select`（如业态选择：住宅/办公/商业/工业/交通枢纽）
  - `type: "multiselect"` → 如 business_type 为多选，需支持多选组件

**`/api/diagnose` 对接**：
- **延迟高（5-10s）**：必须设计加载态。建议全屏遮罩 + 动画 + 分阶段提示：
  - 0-2s："正在解析场站画像..."
  - 2-5s："正在检索相似场站（RAG 引擎）..."
  - 5-8s："正在生成优化建议（DeepSeek v4-pro）..."
- 超时处理：建议设置 Axios timeout = 30s，超时时提示"诊断超时，请重试"。
- 失败重试：提供"重新诊断"按钮。

**后端对接综合评分：9 / 10**

> 接口设计清晰，前端职责单一（纯展示层），仅 diagnose 延迟需前端做体验补偿。

### 3.4 RAG 知识库展示层评审

#### 3.4.1 RAG 数据流回顾

```
用户输入 → 前端 /input
    ↓
POST /api/extract → 后端 LLM 解析为结构化画像
    ↓
POST /api/diagnose → 后端并行执行：
    ├── 算法 Stub：规则计算利用率/收益（is_stub: true）
    └── RAG 引擎：
        1. Kimi Embedding → 查询向量
        2. ChromaDB → Top-5 相似场站
        3. DeepSeek v4-pro → 生成 Markdown 分析报告
    ↓
后端合并为 report 对象 → 前端 /report 渲染
```

#### 3.4.2 前端渲染方案

**算法预测面板（左栏）**：
```
┌─────────────────────────┐
│ 🔧 算法预测（Stub）       │
├─────────────────────────┤
│ 预测利用率：5.2%         │
│ 预测年收益：-8.3 万元     │
│ 置信度：低（0.3）         │
│                         │
│ ⚠️ 此为基于规则的模拟预测，│
│    非真实机器学习模型输出   │
└─────────────────────────┘
```
- 关键设计：`is_stub: true` 和 `confidence: 0.3` 必须在 UI 上**显著提示**，避免用户误解为真实预测。
- 建议用 `Alert` 组件（shadcn/ui 的 Alert 变体 warning）展示免责声明。

**RAG 分析面板（右栏）**：
```
┌─────────────────────────┐
│ 🧠 知识库类比（RAG）      │
├─────────────────────────┤
│ 相似场站 Top 3：          │
│ ┌─────────────────────┐ │
│ │ 小鹏 S4 超快充（南山） │ │
│ │ 利用率：25.91% ↑      │ │
│ │ 日均充电：1,240 度     │ │
│ └─────────────────────┘ │
│ ┌─────────────────────┐ │
│ │ 来福士广场充电站       │ │
│ │ 利用率：1.24% ↓       │ │
│ │ 日均充电：89 度        │ │
│ └─────────────────────┘ │
│                         │
│ LLM 分析：               │
│ [Markdown 渲染区域]      │
└─────────────────────────┘
```

**综合建议区**：
- Markdown 渲染：使用 `react-markdown` 或 shadcn/ui 的 Typography 组件。
- **来源标签高亮**：每条建议需高亮 `[算法预测]` / `[知识库类比]` / `[行业规律推断]` 标签。
  - 建议用不同颜色区分：算法=蓝色，知识库=绿色，推断=灰色。
- **量化数字突出**：报告中的"预计可提升 X 万元/年"等数字，建议用 `<strong>` 或高亮背景突出。

#### 3.4.3 Prompt 规范对前端的约束

根据执行计划中的 Prompt 规范，LLM 输出必须包含：
1. **第一部分：现实评估** — 开头一句直击痛点（"你的场站年收益 X 万元，同类最高 Y 万元..."）
2. **第二部分：潜力空间测算** — 标杆场站 vs 常规场站对比表
3. **第三部分：提升路径建模** — 路径A（成本优化）+ 路径B（效率提升）

**前端需配合**：
- Markdown 渲染需支持表格（`react-markdown` + `remark-gfm` 插件）。
- 三层标题（`## 第一部分` / `## 第二部分` / `## 第三部分`）建议做成锚点导航或折叠面板，方便长报告快速定位。

**RAG 展示层综合评分：9 / 10**

> RAG 完全封装在后端，前端职责清晰（渲染 + 高亮）。唯一注意点是 Markdown 表格渲染需引入 GFM 插件。

## 五、改进建议（按优先级排序）

### P0 — 阻塞性问题（必须解决）

1. **[R2] 禁止 URL 传递 diagnose result**
   - 当前方案中 `/report?result=xxx` 不可行，result 数据量可能超 URL 长度限制。
   - **改为**：diagnose 调用成功后通过 React Context（或 Zustand）内存传递，report 页从全局状态读取。
   - profile 可通过 localStorage 传递（数据量小，且支持刷新恢复）。

2. **[R1+R5] 统一错误处理与加载态**

## 三、状态管理与数据流深化

### 3.1 推荐方案：Zustand 全局 Store

上版评审提到"React Context 或 Zustand"，这里给出明确推荐和完整代码设计。

**选择 Zustand 而非 Context 的理由**：
- Context 的 profile 更新会导致所有消费者重渲染（report 页可能已卸载，无影响，但 enrich 页有循环调用）
- Zustand 支持选择性订阅（`useStore(s => s.profile)`），enrich 每答一题更新 profile 时不会触发无关组件渲染
- Zustand 体积 1KB，API 极简，无 Provider 包裹

```typescript
// frontend/src/store/diagnosisStore.ts
import { create } from 'zustand';
import type { StationProfile, DiagnoseResponse } from '@/types/diagnosis';

interface DiagnosisState {
  // 数据
  profile: StationProfile | null;
  diagnoseResult: DiagnoseResponse | null;

  // 状态
  isDiagnosing: boolean;
  error: string | null;

  // 操作
  setProfile: (profile: StationProfile) => void;
  updateProfileField: (key: string, value: unknown) => void;
  setDiagnoseResult: (result: DiagnoseResponse) => void;
  setIsDiagnosing: (v: boolean) => void;
  setError: (err: string | null) => void;
  reset: () => void;
}

export const useDiagnosisStore = create<DiagnosisState>((set) => ({
  profile: null,
  diagnoseResult: null,
  isDiagnosing: false,
  error: null,

  setProfile: (profile) => set({ profile }),
  updateProfileField: (key, value) =>
    set((state) => ({
      profile: state.profile ? { ...state.profile, [key]: value } : null,
    })),
  setDiagnoseResult: (result) => set({ diagnoseResult: result, isDiagnosing: false }),
  setIsDiagnosing: (v) => set({ isDiagnosing: v }),
  setError: (err) => set({ error: err, isDiagnosing: false }),
  reset: () => set({ profile: null, diagnoseResult: null, isDiagnosing: false, error: null }),
}));
```

### 3.2 三页面数据流（修正版）

```
/input
  ├─ 用户提交 → POST /api/extract → 获取 profile
  ├─ store.setProfile(profile)
  ├─ localStorage.setItem('chargeMind_profile', JSON.stringify(profile))
  └─ navigate('/enrich')

/enrich
  ├─ onMount: 从 store 读取 profile（若无则从 localStorage 恢复）
  ├─ 循环：
  │   ├─ POST /api/enrich(profile) → 获取 nextQuestion
  │   ├─ 用户回答 → store.updateProfileField(key, value)
  │   ├─ localStorage 即时同步
  │   └─ 若 complete=false → 继续循环
  ├─ 若 complete=true：
  │   ├─ store.setIsDiagnosing(true)
  │   ├─ navigate('/report')
  │   └─ POST /api/diagnose(profile) → store.setDiagnoseResult(result)
  └─ 异常：若 diagnose 失败 → store.setError(msg)

/report
  ├─ onMount: 从 store 读取 diagnoseResult
  ├─ 若 isDiagnosing=true → 显示加载动画
  ├─ 若 error 非空 → 显示 ErrorFallback + "重新诊断"按钮
  ├─ 若 result 已就绪 → 渲染双引擎报告
  └─ 若 result 为空 + 非 diagnosing → 显示空状态"请先完成诊断"
```

### 3.3 关键修正：diagnose 不作为 report 页 onMount 的副作用

原始方案中 `POST /api/diagnose` 放在 `/enrich` 完成后触发，而非 `/report` 页内调用。这样：
- `/enrich` 页提交后立即显示"诊断中"过渡态
- `/report` 页只负责读取结果并渲染，无副作用
- 避免 `/report` 页刷新后重复调用 diagnose

## 五、Axios 客户端完整封装

上版评审给出了封装建议，这里补充完整实现，作为开发的直接起点：

```typescript
// frontend/src/lib/api.ts
import axios from 'axios';
import type {
  ExtractRequest, ExtractResponse,
  EnrichResponse,
  DiagnoseResponse,
  StationProfile,
} from '@/types/diagnosis';

const apiClient = axios.create({
  baseURL: '/api',
  timeout: 30000,
  headers: { 'Content-Type': 'application/json' },
});

// 响应拦截器：统一错误处理
apiClient.interceptors.response.use(
  (response) => response,
  (error) => {
    if (error.code === 'ECONNABORTED') {
      throw new Error('请求超时，请检查后端服务是否启动');
    }
    if (error.response) {
      const { status, data } = error.response;
      switch (status) {
        case 422:
          throw new Error(data?.detail || '输入数据格式有误，请检查后重试');
        case 500:
          throw new Error('诊断服务暂时不可用，请稍后重试');
        case 503:
          throw new Error('LLM 服务限流，请等待 30 秒后重试');
        default:
          throw new Error(data?.detail || `服务异常 (${status})`);
      }
    }
    if (error.request) {
      throw new Error('无法连接服务器，请检查网络');
    }
    throw error;
  }
);

// API 方法
export const extractProfile = (userInput: string) =>
  apiClient.post<ExtractResponse>('/extract', { user_input: userInput } as ExtractRequest);

export const enrichProfile = (profile: StationProfile) =>
  apiClient.post<EnrichResponse>('/enrich', { profile });

export const diagnoseStation = (profile: StationProfile) =>
  apiClient.post<DiagnoseResponse>('/diagnose', { profile });

export default apiClient;
```

**关键设计决策**：
- `baseURL: '/api'` —— 依赖 Vite proxy 转发，前端无绝对域名，构建后需 nginx 反向代理
- 超时 30s —— 覆盖 diagnose 最坏 10s + 网络抖动
- 拦截器中 503 单独处理 —— DeepSeek API 可能存在并发限流
