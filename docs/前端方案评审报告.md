### 3.3 与后端接口对接

前端需要调用的后端接口：

| 接口 | 方法 | 用途 | 输入 | 输出 |
|------|------|------|------|------|
| `/api/extract` | POST | 解析用户自然语言描述 | `user_input: string` | `profile: object` |

**评分**：9/10（接口设计清晰，前端职责单一）

### 3.4 RAG 知识库接入

**RAG 的接入路径**：

```
用户输入 → 前端 /input 页面
    ↓
前端 POST /api/extract → 后端解析为结构化画像
    ↓
前端 POST /api/diagnose → 后端执行双引擎诊断
    ↓
【后端内部 RAG 流程】
    1. 调用 rag/retriever.py → 用 Kimi Embedding API 生成查询向量
    2. ChromaDB 向量检索 → 返回 Top-5 相似场站
    3. 调用 DeepSeek v4-pro → LLM 分析相似场站，生成优化建议
    ↓
后端返回 diagnose 结果 → 前端 /report 页面渲染
```

**前端看到的 RAG 输出**：

```json
{
  "rag": {
    "similar_stations": [
      {"station_id": "xxx", "similarity_score": 0.84, "metadata": {...}},
      ...
    ],
    "analysis": "Markdown 格式的 LLM 分析报告"
  },
  "report": {
    "executive_summary": "一句话总结",
    "recommendations": [
      {"title": "...", "source": "[知识库类比]", "detail": "..."}
    ]
  }
}
```

**前端渲染方式**：
- 相似场站 → 卡片列表展示（名称、利用率、日均充电量）
- LLM 分析 → Markdown 渲染
- 优化建议 → 带来源标签的列表

**评分**：9/10（RAG 完全封装在后端，前端只需渲染，职责边界清晰）

## 五、改进建议（按优先级排序）

### P0（必须做）
1. ** diagnose 加载状态**：接口慢，必须加"诊断中..."动画，避免用户以为卡死
2. **错误降级**：API 失败时显示友好提示，不要白屏

### P1（建议做）
3. **localStorage 缓存**：问卷填写到一半刷新页面不丢失进度
4. **示例数据一键填入**：降低用户首次使用门槛

### P2（可选优化）
5. **PWA 支持**：后续 2 小时可追加，添加到手机桌面
6. **流式输出**：LLM 报告逐字显示，增强"AI 实时思考"感
