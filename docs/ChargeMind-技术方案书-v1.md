> 项目：ChargeMind 充电场站智能诊断平台（黑客松 Demo 版）  
> 日期：2026-04-24  
> 版本：v1.0

---

## 一、项目概述

### 1.1 项目背景

ChargeMind 是一个面向充电场站运营商的智能诊断平台。核心痛点：运营商投入大量资金建设充电场站，但缺乏数据驱动的运营优化能力，导致利用率低、收益差。

### 1.2 产品定位

**黑客松 Demo 目标**：用户输入场站描述 → 系统引导补充信息 → 生成一份包含"算法预测 + 知识库类比"双引擎的诊断报告，给出具体的运营优化建议。

### 1.3 核心差异化

- **不是单纯的 LLM 聊天**：基于真实的深圳 10,942 个充电场站数据构建 RAG 知识库
- **双引擎交叉验证**：算法 Stub（基于规则的预测）+ RAG（相似场站检索 + LLM 分析）
- **数据可追溯**：每条建议标注来源，是算法算出来的还是知识库类比出来的

## 二、系统架构

### 2.1 整体架构

```
┌─────────────────────────────────────────────────────────────────┐
│                         前端层（Web）                            │
│   React 18 + TypeScript + Tailwind CSS + shadcn/ui              │
│   ┌──────────────┐  ┌──────────────┐  ┌──────────────────────┐  │
│   │ /input 输入页 │→│ /enrich 问卷页│→│ /report 诊断报告页    │  │
│   └──────────────┘  └──────────────┘  └──────────────────────┘  │
└─────────────────────────────────────────────────────────────────┘
                              │ HTTP
                              ▼
┌─────────────────────────────────────────────────────────────────┐
│                       后端层（FastAPI）                          │
│                                                                  │
│   POST /api/extract    → LLM 解析用户输入为结构化画像            │
│   POST /api/enrich     → 判断缺失字段，生成追问问题              │
│   POST /api/diagnose   → 双引擎并行诊断                         │
│                         ├── 算法 Stub（基于规则的预测）           │
│                         └── RAG 引擎（相似场站检索 + LLM 分析）   │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
                              │
            ┌─────────────────┼─────────────────┐
            ▼                 ▼                 ▼
   ┌──────────────┐  ┌─────────────────┐  ┌──────────────┐
   │  ChromaDB    │  │  Kimi API       │  │  DeepSeek API│
   │  本地向量库   │  │  Embedding      │  │  Chat        │
   │  10,942条    │  │  生成向量       │  │  生成分析    │
   └──────────────┘  └─────────────────┘  └──────────────┘
```

### 2.2 前端层

**技术选型**：
- **React 18**：UI 框架，组件化开发
- **TypeScript**：类型安全，减少运行时错误
- **Vite**：构建工具，启动快、HMR 快
- **Tailwind CSS**：原子化 CSS，不写样式文件，暗色主题原生支持
- **shadcn/ui**：基于 Radix UI 的组件库，暗色主题默认好看，开箱即用
- **React Router**：三页面路由（/input → /enrich → /report）
- **Axios**：HTTP 客户端，统一 API 调用

**三个页面**：
1. **输入页（/input）**：用户输入场站自然语言描述，系统调用 `/api/extract` 解析为结构化数据
2. **问卷页（/enrich）**：系统根据缺失字段逐步追问，用户补充关键信息（区域、业态、装机功率、桩数等）
3. **报告页（/report）**：展示双引擎诊断结果——算法预测面板 + RAG 知识库分析面板 + 综合优化建议

### 2.3 后端层

**框架**：FastAPI（Python 3.9）

**三个核心接口**：

| 接口 | 方法 | 功能 | 说明 |
|------|------|------|------|
| `/api/extract` | POST | LLM 解析用户输入 | 将自然语言描述解析为结构化场站画像 |
| `/api/enrich` | POST | 追问缺失字段 | 判断画像中缺失的关键字段，返回下一个问题 |
| `/api/diagnose` | POST | 双引擎并行诊断 | 算法 Stub + RAG 同时运行，返回综合报告 |

**算法 Stub**：
- 基于规则的预测模型（非真实机器学习模型）
- 公式：利用率 = 区域平均利用率 × 业态系数 × 规模系数
- 输出：预测利用率、预测年收益、置信度
- 明确标记 `is_stub: true`，提示用户此为模拟预测

**RAG 引擎**：
- 检索：根据用户场站画像，在 ChromaDB 中检索 Top-5 相似场站
- 分析：将相似场站数据喂给 DeepSeek v4-pro，生成优化建议
- 输出：相似场站列表 + LLM 分析报告

### 2.4 数据层

**ChromaDB 本地向量库**：
- 存储：10,942 条深圳充电场站的向量数据
- 大小：约 78MB（已提交 git，团队成员无需重建）
- 索引字段：场站名称、区域、业态、装机功率、利用率、日均充电量、高峰时段等

## 三、RAG 详细设计

### 3.1 为什么需要 RAG

如果没有 RAG，LLM 只能基于"行业常识"泛泛而谈（如"办公区场站利用率一般在 3-7%"）。

**有了 RAG**，LLM 能看到：
- "和你最像的 5 个场站中，利用率从 0.45% 到 25.91%，差距 57 倍"
- "小鹏 S4 超快充在同一区域做到 25.91%，是因为抓住了午间 13:00 高峰"
- "来福士广场利用率只有 1.24%，可能是因为停车费高"

**结论从"泛泛而谈"变成"基于真实数据的具体建议"。**

### 3.2 RAG 流程

```
用户场站画像
    ↓
【Embedding】Kimi API 将查询文本转成 1024 维向量
    ↓
【检索】ChromaDB 向量相似度搜索 → Top-5 相似场站
    ↓
【分析】DeepSeek v4-pro 读取相似场站数据 → 生成优化建议
    ↓
返回：相似场站列表 + 分析报告
```

### 3.3 检索策略

**三层漏斗**：
1. **精确匹配**：同区域 + 同业态
2. **放宽业态**：同区域 + 近似业态（如果精确匹配结果不足）
3. **全市范围**：同业态（如果区域匹配结果不足）

**给 LLM 的输入格式**：
- 不是文字段落，而是一张**结构化对比表**
- 包含：装机功率、桩数、利用率、日均充电量、高峰时段、快充占比等
- LLM 像专家一样读表格，找出规律

### 3.4 Prompt 设计（报告生成）

**三层结构**（基于嘉老师需求定义）：

**第一部分：现实评估**
- 估算全年充电量和全年收益
- 与相似场站对比定位
- 开头一句直击痛点："你的场站年收益 X 万，同类最高 Y 万，差距 W 万"

**第二部分：潜力空间测算**
- 把相似场站分为"标杆场站"（高利用率）和"常规场站"（低利用率）
- 分别提炼两组的共同特征
- 量化收益差距
- 分析外部因素（周边竞争、车队行为、停驶习惯）

**第三部分：提升路径建模**
- 路径 A：成本优化（电力现货、补充协议）
- 路径 B：效率提升（引流机制、峰谷优化）
- 按"投入少见效快 → 投入大长期见效"排序

**约束**：
- 不考虑人员成本、运维成本
- 所有数字标注来源：`[算法预测]` / `[知识库类比]` / `[行业规律推断]`
- 收益差距必须量化（不说"有较大提升空间"，说"预计可提升 X 万元/年"）

## 四、双 API 策略

### 4.1 为什么用两个 API

| 功能 | 服务 | Base URL | 原因 |
|------|------|---------|------|
| **Embedding** | Kimi | `api.kimi.com/coding/v1` | 用户的 Kimi 699 套餐支持 Embedding，且已实测可用 |
| **Chat** | DeepSeek | `api.deepseek.com` | Kimi 套餐不支持 Chat Completion（403），DeepSeek v4-pro 支持且推理能力强 |

### 4.2 各自职责

- **Kimi Embedding**：把"南山科技园办公区充电站，装机 500kW"变成 1024 维向量，用于相似度检索
- **DeepSeek Chat**：读取相似场站数据，分析成功/失败模式，生成优化建议报告

## 五、接口定义

### 5.1 POST /api/extract

**请求**：
```json
{"user_input": "我在深圳南山区科技园，有15个快充桩，周边主要是写字楼"}
```

**响应**：
```json
{
  "profile": {
    "region": "南山区",
    "business_type": ["办公区"],
    "total_installed_power": 500,
    "pile_count": 15
  },
  "llm_used": true
}
```

### 5.2 POST /api/enrich

**请求**：
```json
{"profile": {"region": "南山区", "business_type": ["办公区"]}}
```

**响应**：
```json
{
  "complete": false,
  "next_question": {
    "key": "total_installed_power",
    "question": "装机总功率大约多少 kW？",
    "type": "number"
  },
  "missing_count": 5
}
```

### 5.3 POST /api/diagnose

**请求**：
```json
{"profile": {"region": "南山区", "business_type": ["办公区"], "total_installed_power": 500, "pile_count": 15}}
```

**响应**：
```json
{
  "profile": {...},
  "algorithm": {
    "predicted_utilization": 0.052,
    "annual_profit": -83000,
    "confidence": 0.3,
    "is_stub": true
  },
  "rag": {
    "similar_stations": [
      {"station_id": "xxx", "similarity_score": 0.84, "metadata": {...}}
    ],
    "analysis": "Markdown 格式的 LLM 分析报告"
  },
  "report": {
    "executive_summary": "...",
    "recommendations": [
      {"title": "...", "source": "[知识库类比]", "detail": "..."}
    ]
  }
}
```

## 七、前端与后端的交互流程

```
用户打开 /input
    ↓
输入描述 → POST /api/extract
    ↓
获取 profile → 跳转到 /enrich
    ↓
循环：
  POST /api/enrich (profile) → 获取 next_question
  用户回答 → 更新 profile
直到 complete=true
    ↓
POST /api/diagnose (profile) → 获取 report
    ↓
跳转到 /report
    ↓
页面渲染：
  - 算法预测面板（利用率、年利润）
  - RAG 分析面板（相似场站、优化建议）
  - 综合建议区（Markdown + 来源标签）
```
